几种取模优化方法

发表于 2023-01-06

斗胆续写《几种取模优化方法（译自 min-25 的博客）》。

用来测试常数的计算任务是 $p\gets p\times p$，重复 $10^8$ 次，并对某个数取模。下表是各种优化方式的平均用时，单位为毫秒。

优化方式 32 位模数 64 位模数
编译器优化 423(338) 2742(2662)
不优化 1259(1000) 2691(2662)
浮点数除法 983 1182
Barrett Reduction 433 641
Montgomery Multiplication 338 310

测试环境：SOJ，C++14，默认 O2。只有 Barrett Reduction 的代码用了一些先进的语法，于是开 C++17。

32 位模数在 $10^8+7\sim 10^9+7$ 选取。64 位模数在 $10^{17}+3\sim 10^{18}+3$ 选取。质数合数都有。

更新：我一开始没考虑 signed 和 unsigned 类型的区别。《A modulo multiplication method that is 2x faster than compiler implementation》 指出不考虑符号的取模会快一些。Barrett Reduction 和 Montgomery Multiplication 的实现都是无符号计算。“编译器优化”和“不优化”用相应 unsigned 类型测试得到的用时放在上表的括号内。

编译器优化

把模数用 constexpr 修饰。

32 位模数编译器实现了 Barrett Reduction（可以看汇编），64 位模数编译器似乎不优化。

测试 64 位模数的时候，把 int 改成 long long，long long 改成 __int128。

浮点数除法

名字是我起的。看到有人这么写（64 位模数）：

首先这是未定义行为，因为 p*p 很可能发生有符号整型溢出。但是减号右边也会溢出，只要两边的差不会溢出就行。然后浮点数有精度误差，所以运算结果可能比 0 小一点，也可能比模数大一点。在我选取的 64 位模数中，如果模数是 $m$，则 $10^8$ 次运算结果均位于 $-0.1m\sim 1.01m$。所以在输出的时候用 (p%mod+mod)%mod 就没事了。暂时没发现正确性问题。

虽然 SOJ 上 long double 占用 16 字节，但是只有 80 位精度。其中尾数有 64 位，刚好够精确表示一个 long long。这说明 (long double)(p)*p/mod 的绝对误差不会超过 ±1。
