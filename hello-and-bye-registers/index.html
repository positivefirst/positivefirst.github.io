这道题最幽默的地方在于参考代码里的 next 函数调用 $2n$ 次已经超时了。每一个基本操作都依赖上一步的结果，并行不了一点。注意到加法具有交换律，不需要按顺序调用 query。所以对 $n$ 分段，拿矩阵快速幂先跳过整段的 next 调用，获得一些可以独立递推的 seed。然后还是写暴力，循环展开。

能不能再快一点呢。现在段与段之间完全独立，可以并行计算。QOJ 支持 avx512 指令集，一口气递推 $8$ 个 seed。但我没有找到 pdep 指令的 SIMD 版本，只好逐一调用 query。为减少数据依赖，我开了 unsigned ans_part[8] 累加答案。诡异的是，如果没有循环展开，ans_part 可以被优化成 $8$ 个通用寄存器；如果循环展开，ans_part 反而退化为栈上的数组，读写都要额外指令。循环展开怎么可能占用更多的通用寄存器呢？明明我把（没循环展开的）循环体汇编代码复制几遍就该能跑。也不知道 GCC 怎么想的，反正开 O1~O3 一个样。

于是我就有了内嵌汇编的想法。顺便把语句重排一下，解决数据依赖和端口冲突。（GCC 也会重排，但我看着像局部 random shuffle。）感觉把向量指令和标量指令均匀混合就挺不错。打算把汇编塞进去了，却发现比想象中要麻烦，GCC 那套汇编语法跟 Compiler Explorer 还有点区别。开头提到的 register 本来作用应该是为内嵌汇编操作寄存器提供方便，否则 asm 指定的寄存器名字没意义。虽然最终没有内嵌汇编，但我解决了内嵌汇编前遇到的问题，可以让 GCC 把 ans_part 塞进寄存器了。只是聪明的 GCC shuffle 了我精心重排的语句，速度应该不如内嵌汇编。

向量指令集也有代价，虽然批处理快，但提取单个元素（放入通用寄存器）比较麻烦。我发现 GCC 有两招。第一招类似分治，从 zmm 中提取 ymm，从 ymm 中提取 xmm，最后把 xmm 中的元素放入通用寄存器。由于 zmm、ymm、xmm 的包含关系，提取 zmm、ymm 的低位不消耗时间。第二招是用栈空间中转，一条指令把 zmm 直接写到栈上，栈上的东西就可以像普通数组一样访问了。这意味着寄存器要和内存（这里显然能进 L1 缓存）交换数据。虽然通信延迟比寄存器之间的转移延迟大，但吞吐量也大，而且用的端口不容易和别的指令冲突。如果能提前发出读/写内存的指令，或者说解决数据依赖，理论上第二招更快。但我没有严格测试，因为 GCC 知道这两招等价，便能随心所欲地改变实现方式。
